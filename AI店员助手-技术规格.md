# 剧本杀AI店员助手 — 技术规格文档

> 版本: v1.9 | 日期: 2026-04-29 | 状态: 最终定稿，可作为正式开发输入文档

---

## 1. 产品定位

面向剧本杀连锁店/直营店的**内部AI知识助手**。让员工随时查询任意剧本内容、让管理员高效管理和改编剧本库、让排班等日常事务查询自动化。

**不是**创作工具（那是诡案工坊的定位），而是**剧本知识管理与运营辅助**工具——包括剧本问答检索、剧本改编（改细节/调视角/换背景，不改核心诡计）、以及员工信息管理。

**核心设计原则：** 剧本杀的核心是诡计和逻辑闭环——凶手可能是表层A、深层B、实际C。只看片段会被局部信息误导，**必须全本上下文才能正确回答**。因此所有剧本问答均走全本读取，不做向量切块检索。

## 2. 系统架构

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     员工 / 管理员浏览器                       │
│                   （Next.js 14 Web 应用）                    │
└────────────────────────┬────────────────────────────────────┘
                         │ 内网 HTTPS（自签名证书）
┌────────────────────────▼────────────────────────────────────┐
│                     Next.js API Routes                        │
│          （对话 / 剧本读取 / 排班 / 改编 / 文件服务）          │
└──────┬──────────────┬────────────────────────────┬───────────┘
       │              │                            │
       ▼              ▼                            ▼
  ┌──────────┐  ┌────────────────┐       ┌──────────────────────┐
  │PostgreSQL│  │  文件存储       │       │  FastAPI 微服务      │
  │员工/排班 │  │ /data/scripts/  │       │  （文档解析）        │
  │          │  │  DOCX / PDF    │       │  python-docx         │
  │          │  │  + 解析缓存    │       │  PyMuPDF             │
  └──────────┘  └────────────────┘       └──────────┬───────────┘
                                                    │
                                              ┌──────────────────────────────────┐
                                              │        DeepSeek 云端 API          │
                                              │                                  │
                                              │  主模型：deepseek-v4-pro          │
                                              │  备模型：deepseek-v4-flash       │
                                              │  （管理员手动切换）               │
                                              └──────────────────────────────────┘
```

**架构简化说明（相比 v1.0-draft）：**
- ❌ 去掉 Qdrant 向量数据库
- ❌ 去掉 Embedding 模型（DeepSeek 官方无独立 Embedding API）
- ✅ 所有问答和改编均基于全本上下文
- ✅ 简化后服务数量减少，运维成本降低

### 2.2 后端技术栈

**选型结论：Next.js 全栈为主 + FastAPI 微服务（仅文档解析）**

| 模块 | 技术选型 | 理由 |
|------|---------|------|
| 主后端 | **Next.js API Routes** | 复用现有技术栈，单一部署单元 |
| 文档解析 | **FastAPI 微服务**（独立容器） | python-docx / PyMuPDF 无法 Node.js 调用 |
| LLM 调用 | **Next.js API Routes 直调** | 无需中间层，直接调用 DeepSeek API |
| 剧本缓存 | **本地磁盘文件** | `/data/scripts/{id}/parsed.txt`，无额外依赖 |
| 员工数据库 | **PostgreSQL**（通过 Next.js API 调用） | 权限成熟，和 Next.js 部署同机器 |

**FastAPI 微服务职责（唯一职责）：**
- 接收 DOCX/PDF 文件 → 解析提取结构化纯文本 → 返回
- 内存占用 < 500MB，极轻量

### 2.3 LLM 模型配置

**双模型策略：手动切换，管理员自主决策**

> **v1 设计选择：** 接受手动切换的停机风险——主模型不可用时，系统提示员工"AI 服务暂时不可用，请稍后重试"，管理员收到通知后手动切换模型。暂不实现自动降级，理由：内网应用可用性要求低于 7×24 公共服务，且自动降级逻辑（检测失败 → 切换模型 → 验证 → 恢复）增加复杂度。待 v2 再引入自动故障转移。

| 角色 | 模型 | 用途 |
|------|------|------|
| **主模型** | `deepseek-v4-pro` | 日常问答、改编生成（主力） |
| **备用模型** | `deepseek-v4-flash` | 主模型不可用时管理员手动切换 |

**切换机制：**
- 管理员在后台设置面板中选择当前激活模型
- 切换后实时生效，无需重启服务
- 前端显示当前使用模型名称
- 主模型不可用时，前端显示醒目提示"AI 服务不可用，管理员正在处理"

**模型选择逻辑（Next.js API Routes）：**

```typescript

const activeModel = await getActiveModel(); // 'deepseek-v4-pro' | 'deepseek-v4-flash'

// 注：以下为示意代码，完整实现需补充 POST method、headers、body 等

  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${process.env.DEEPSEEK_API_KEY}`
  },
  body: JSON.stringify({
    model: activeModel,  // 动态切换
    messages: [...]
  })
})
```

**配置表（PostgreSQL）：**

```sql
app_settings (
  id VARCHAR(50) PRIMARY KEY,
  active_llm_model VARCHAR(50) DEFAULT 'deepseek-v4-pro',
  -- 其他设置...

```

**API 并发控制（Next.js API Routes）：**

v1 采用简单的请求信号量控制并发上限，防止多员工同时提问时触发 DeepSeek API 429 限流：

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大并发 LLM 请求数 | 5 | 超过的请求排队等待 |
| 排队超时 | 60 秒 | 超出则返回"AI 服务繁忙，请稍后重试" |
| 实现方式 | 内存信号量（`async-semaphore` 或自定义 Promise 队列） | v1 简单方案；v2 可升级为 Redis 分布式锁 |

```typescript
// 示意：并发控制
// 正确逻辑：判断"正在执行的请求数"，而非"队列长度"
// 此为简化示意代码，生产环境建议使用 async-mutex 或自定义 Promise 队列实现
const running = { count: 0 };
const MAX_CONCURRENT = 5;

async function withSemaphore<T>(fn: () => Promise<T>): Promise<T> {
  if (running.count < MAX_CONCURRENT) {
    running.count++;
    try {
      return await fn();
    } finally {
      running.count--;
      // 唤醒队首等待的请求
    }
  }
  // 超过并发上限，排队等待（实际实现需补充完整排队/出队/唤醒逻辑）
  const waitingQueue: Array<() => void> = [];
  return new Promise<T>((resolve) => {
    // waitingQueue.push(() => { resolve(fn()); });
  });
}
```

> **429 限流处理：** 若 DeepSeek API 返回 429，后端自动等待 5 秒后重试 1 次；重试仍失败则记录 `operation_logs`，前端提示"AI 服务暂时不可用，管理员已收到通知"。

## 3. 剧本问答核心原理

### 3.1 为什么必须全本读取

剧本杀的逻辑闭环特性决定切片检索不可靠：

```
员工问：「这个本谁是凶手？」

❌ 切片检索的问题：
  召回片段A：「第二幕A说B是凶手」
  召回片段B：「第三幕B说自己是无辜的」
  召回片段C：「第四幕C说A在撒谎」
  → AI可能给出一个局部答案，被某个片段误导

✅ 全本读取的优势：
  AI 看到：
  - 表层：A被多人指控
  - 中层：B有作案动机和机会
  - 深层：线索指向C
  - 真相：核心诡计决定了真正凶手是D
  → AI能结合全本逻辑给出正确且有依据的回答
```

### 3.2 剧本字数与上下文评估

| 剧本类型 | 幕数 | 角色数 | 总字数 | Token估算 | 占100万比例 |
|---------|------|--------|--------|----------|------------|
| 标准6人本 | 5幕 | 6人 | 7,500-10,000 | ~20,000 | 2% |
| 典型8人本 | 6幕 | 8人 | 12,000-15,000 | ~30,000 | 3% |
| 大体量10人+ | 7-8幕 | 10人+ | 17,500-28,000 | ~50,000 | 5% |
| 超长城限本 | 8-10幕 | 12人+ | 28,000-50,000 | ~80,000 | 8% |

> **结论：`deepseek-v4-pro` / `deepseek-v4-flash` 的 100万 Token 上下文可覆盖市面所有剧本。** 即使最大的城限本，全本读取后仍留有 92% 空间给 Question + Prompt + Answer。

### 3.3 缓存策略与 Token 计算

缓存分为两层，各司其职：

**第一层：解析缓存**（节省 FastAPI 解析时间和网络带宽）
- 剧本 DOCX/PDF 解析为纯文本后存入 `/data/scripts/{id}/parsed.txt`
- 命中后无需重复调用 FastAPI，节省解析时间（每次 ~1-3 秒）

**第二层：LLM Prompt 组装策略**（控制 Token 消耗）

```typescript
// 每次问答的 messages 组装方式
const messages = [
  { role: 'system', content: SYSTEM_PROMPT },        // 固定系统指令
  { role: 'system', content: scriptCache },            // 剧本全文（仅首次或手动刷新）
  ...conversationHistory.slice(-10),                   // 最近10轮对话（不含剧本）
  { role: 'user', content: currentQuestion }         // 本次问题
]
```

**关键设计：剧本全文只注入一次**——首次问答时将剧本全文作为 system message 发送，后续对话只携带最近 10 轮问答历史（不含剧本）。对话历史按轮压缩精简（每轮保留用户问题 + AI 回答的摘要版），进一步控制 Token。

**Token 消耗计算：**

| 场景 | 输入 Token | 输出 Token | 说明 |
|------|-----------|----------|------|
| 首次问答（新剧本） | 剧本 ~30K + Prompt ~1K + 对话历史 ~0 | ~500-2K | 剧本全文注入 |
| 后续问答（同一剧本） | 对话历史 ~5K + Prompt ~1K + 问题 ~200 | ~500-2K | 无需重复发剧本 |
| 改编（全本生成） | 剧本 ~30K + 指令 ~500 | ~20-50K | 全本输出 |

**Token 节省效果（基于日常使用场景）：**

> **假设前提：** 剧本库规模100本，每本平均被问5次，共500次/天（100人·次/天）。
> - 首次问答：每本首次被问时发送全本，约31K Token/次 × 100次/天 = 3.1M Token/天
> - 后续问答：同一剧本重复被问，6K Token/次 × 400次/天 = 2.4M Token/天
> - 合计：~5.5M Token/天 → ~165M Token/月
> - 月成本（v4-flash，标准价）：¥62/月；月成本（优惠价）：¥15-20/月

相比无缓存策略（全本重复发送）：
  节省率：~60-70%

> **注意：** 缓存节省的是 FastAPI 解析时间和 LLM 重复发送剧本的 Token，而非绕过 LLM 调用本身。每次问答的对话历史会随轮次累积，但通过摘要压缩控制在合理范围内。

**对话历史压缩策略（后端执行）：**

当同一剧本对话轮次超过 10 轮时，后端在组装 messages 时对超出部分做摘要压缩：

| 参数 | 值 | 说明 |
|------|-----|------|
| 保留轮次 | 最近 10 轮（原始） | 用户问题和 AI 回答保留原文 |
| 压缩轮次 | 第 11 轮起 | 调用 LLM 生成摘要，每轮摘要 ≤ 300 Token |
| 压缩触发时机 | 对话总轮次 > 10 | 不每轮压缩，只在超阈值时批量压缩一次 |
| 压缩执行位置 | Next.js API Routes 后端 | 不在前端执行，避免 Token 浪费 |

```typescript
// 压缩逻辑示意
if (conversationRounds > 10) {
  const recentRounds = history.slice(-10);  // 保留最近10轮原文
  const olderRounds = history.slice(0, -10); // 超出部分做摘要
  const compressedOlder = await llmSummarize(olderRounds); // 摘要 ≤ 300 Token
  history = [...compressedOlder, ...recentRounds];
}
```

> **Token 成本注意：** 摘要压缩本身也消耗 Token（约 300 Token/次压缩）。但这笔开销远小于无压缩时重复发送全本剧本的代价。以 20 轮对话为例：无压缩 = 10轮 × 额外Token；有压缩 = 摘要1次（300 Token）+ 保留10轮，仍能节省约 60-70%。

**缓存失效与健康检查：**
- 管理员上传新版本剧本文件 → 自动清除旧缓存
- 管理员手动触发"重新解析"
- 缓存文件损坏检测：读取时计算校验和（MD5），与入库时记录的值对比；不一致则自动重新解析
- 缓存无过期机制（剧本内容不变，缓存永久有效）
- 磁盘空间告警：磁盘使用率超过 80% 时前端提醒管理员，可选择性清除长期未访问的剧本缓存

## 4. 三大核心模块

### 4.1 模块一：剧本检索问答

**适用角色：** 员工 + 管理员

**对话流程：**

```
① 用户选择剧本（或 AI 自动识别剧本名）
      │
      ▼
② 检查本地缓存
      │
      ├── 命中 → 读取缓存文本
      └── 未命中 → FastAPI 解析 → 写缓存
      │
      ▼
③ 组装 Prompt
   【系统指令】你是剧本杀专家，只基于以下剧本内容回答...
   【剧本全文】
   【用户问题】
      │
      ▼
④ deepseek-v4-pro（当前激活模型）生成回答（流式输出）
      │
      ▼
⑤ 前端展示 + 会话上下文保留（最近 10 轮）
```

**Prompt 设计要点：**
- 强制要求「只基于剧本内容回答」，不确定时明确说「剧本中没有明确提及」
- 回答涉及凶手/诡计时，需要给出逻辑推理过程（而非只给结论）
- 支持追问，保持会话上下文

**Agent System Prompt 草案（v1 初版 · 修订版）：**

```
你是剧本杀店铺的AI助手，主要职责是**剧本知识查询**，辅助职责是**剧本改编**（仅管理员触发）。

【身份与职责】

- 查询时：严格基于已入库剧本内容回答
- 运营类问题（DM技巧、开场白、适用人群判断）：可基于通用知识回答，标注「以下为通用建议，请结合实际情况调整」
- 跨剧本对比：只限已入库剧本范围内
- 你的用户是**DM和员工**，不是玩家——始终从**主持者视角**出发，而非玩家视角

【行为准则】

- 剧本内容问题：基于剧本全文，不猜测、不脑补、不捏造原文没有的内容
- 不确定时：「剧本中没有明确提及相关内容」
- 不回答与剧本杀运营无关的问题（如闲聊、政治话题、个人生活建议）
- 涉及敏感内容（真实历史、暴力等）：只陈述剧本中的事实描述，不做价值判断；角色动机分析需基于剧本中的线索和设定进行逻辑推导，不做超出剧本的情感渲染

【语气风格】

- 语言：中文，口语化但不随意，专业、干脆、不啰嗦
- 长度：普通问题控制在 200 字以内；推理类问题可适当展开
- 禁止：使用 emoji，使用"亲""宝"等网络用语，使用"作为一个AI"等自我指称

【回答格式】

- 普通问题：直接回答，语言简洁专业
- 凶手/诡计类问题：**先展示关键证据和推理过程**（引用幕数/线索编号），最后给出结论；如果员工追问"直接告诉我答案"，再给出简洁结论
- 结构类问题：按**幕数顺序**组织回答，每幕简述关键事件和涉及角色；如问特定幕，按**事件时间线**描述，标注每个事件出自哪个角色
- **引用格式（统一）**：`【第X幕 · 角色名】"原文摘录"` 或 `【线索卡#编号】"原文摘录"`
  - 示例：根据【第三幕 · 角色B】"我在仓库发现了带血的手套"，可以推断...

【边界处理】

- 未指定剧本时（用户只问"谁是凶手"而未提剧本名）：「请问你想了解哪个剧本？以下是最近使用或已入库的剧本：...」（展示剧本列表）
- 剧本名不存在时：「未找到该剧本，请确认名称是否正确。当前已入库的剧本有：...」
- 同时提及多个剧本时：「一次只能查询一个剧本，请选择其中一个：...」
- 问题超出剧本内容范围：「剧本中没有明确提及。建议查看该剧本的DM手册或咨询有经验的同事。」——不提及"联系发行方"
- 问题涉及剧本适用人群/难度等运营判断：基于已有的剧本标签（幕数、角色数、类型、难度）给出参考意见
- 剧本内容矛盾（如角色A声称未去过某地，但线索显示其指纹出现）：如实指出矛盾，分别引用矛盾的两处原文，不做裁决。回复格式：「注意：第二幕角色A声称未去过仓库（【第二幕·A】"..."），但线索X显示A的指纹出现在仓库（【线索X】"..."）。这可能是剧本设计的误导，也可能是笔误，建议DM确认。」
- 越权问题（如询问其他员工信息）：「我没有权限查询该信息，建议联系管理员」
- 简单寒暄（你好/早上好等）：简短回应后引导提问，如「你好！有什么剧本相关的问题需要帮忙吗？」
- 无关闲聊（政治/八卦等）：「我专注于剧本相关的问题，有其他需要帮忙的吗？」

【多剧本切换】

- 用户切换到另一剧本时，提醒：「已切换到《新剧本名》，之前的对话上下文已清除。」
- 不同剧本的对话上下文互相隔离，避免信息混淆

【关键事实的不确定处理】

以下情况在回答末尾附加核实建议（不是所有回答都加）：
- 剧本中存在矛盾线索（已在"内容矛盾"段落处理）
- 凶手身份在剧本中有多层嵌套（表层→深层→真相），推理链超过3步
- 线索依赖角色的主观证词而非客观物证

普通问题（可直接从剧本明确推断的事实）不需要附加核实建议。

【改编场景（管理员专用）】

**输出格式（两层）：**
1. 改编概要（必显示）：3-5句话概括改编范围和主要改动
2. 详细改动表（可展开，按模块分类）：角色改动 / 线索改动 / 场景改动 / 台词改动

**改编分级处理：**
- **禁止（Agent 拒绝执行）：** 删除关键角色、改变核心诡计类型（如密室→开放场景）、删除所有线索指向
  - 遇到红线指令：「这个改编可能破坏剧本的核心推理逻辑，建议重新调整改编范围。」
- **需确认（提示管理员影响）：** 增减幕数、合并/拆分幕（如"让第三幕冲突提前到第二幕"）
  - 遇到此类指令：「注意：此操作可能影响剧情节奏和角色戏份分布，建议确认改动范围后重新输入。」
- **允许：** 元素替换、台词调整、视角扩展、场景细节修改

**改编自检清单（Agent生成后自动核对）：**
- [ ] 角色数量 = 原版
- [ ] 幕数 = 原版（除非指令明确要求增减）
- [ ] 每个角色都有对应的剧本内容（无遗漏）
- [ ] 所有线索卡都有对应内容
- [ ] DM手册与改编后的剧情一致
- [ ] 核心诡计推理链完整（A→B→C→真凶）
- [ ] 改编后的背景设定与剧本内的时间线/场景描述一致（如时代替换后，道具、通讯方式等细节同步更新）

**改编完整正文：** 保持原有幕数/角色数结构，输出完整改编版本

**改编预览前确认环节：**
- 收到改编指令后，AI 先输出**改编方案确认**（3-5句话描述计划做什么改动）
- 示例：「计划将死亡方式从刀杀改为毒杀，保持凶手动机和嫌疑结构不变，线索中的刀相关证据改为毒相关证据。」
- **"需确认"级指令的处理：** 在方案确认环节中一并展示影响说明，示例：「注意：此操作可能影响剧情节奏和角色戏份分布，建议确认改动范围。」管理员确认后继续，不额外增加确认步骤。
- 管理员确认后，AI 再生成全本正文
- 方案确认环节约消耗 31K Token（需注入剧本全文供 LLM 理解上下文），但可避免因指令误解导致的全本重生成（50K+ Token 浪费）
```

**典型问答场景：**

| 员工问 | AI 答 | 备注 |
|--------|-------|------|
| "《xxx》谁是凶手？" | "根据【线索3】和【第四幕·角色B】的证词，关键证据指向C……**结论：C是凶手**" | 先证据推理，后结论；避免直接剧透 |
| "第三幕发生了什么？" | "第三幕：①角色B暴露了……②角色C转移话题……" | 按幕数结构，标注角色 |
| "玩家问到XX线索我该怎么引导？" | "【DM建议】这是通用引导技巧，请根据实际情况调整……" | 标注通用建议，非剧本内容 |
| "线索4怎么用？" | "线索4是关键物证，需要玩家在第三轮搜证才能获得..." | 结合剧本结构回答 |
| "DM手册怎么引导玩家？" | "DM手册建议在此处让玩家投票，通过机制..." | 读取DM手册内容 |

### 4.2 模块二：剧本改编工坊

**适用角色：** 仅管理员

#### 4.2.1 全本读取（仅管理员触发）

改编必须读取完整剧本结构，不依赖切片检索：

```
管理员选择剧本
    │
    ▼
FastAPI 解析 DOCX 原始文件（保留 XML 结构信息）
    │
    ▼
读取完整剧本内容：
  → 所有幕、所有角色的完整剧本正文
  → DM手册原文
  → 线索卡内容
  → 版面格式信息（标题层级、加粗、分段）
    │
    ▼
全本内容 + 改编指令 → deepseek-v4-pro（当前激活模型）
    │
    ▼
生成改编版本 + 改编说明文档
```

#### 4.2.2 改编类型（v1 支持）

| 改编类型 | 描述 | 例子 |
|---------|------|------|
| **元素替换** | 改具体细节，保持结构 | 刀杀→毒杀；民国→现代；角色A→角色B |
| **剧情微调** | 改逻辑衔接，不改结构 | "让第三幕冲突提前到第二幕" |
| **视角扩展** | 以另一角色视角重写 | 为凶手写一个"知情者版本" |
| **手册改编** | 改DM手册措辞/格式 | 把"直接告诉玩家"改成"暗示引导" |

#### 4.2.3 改编流程

```
① 管理员选择剧本
      │
      ▼
② FastAPI 全本解析（模式 B：完整读取）
      │
      ▼
③ 管理员输入改编指令
   「把死亡方式从刀杀改成毒杀，凶手动机不变，
     线索中刀相关证据改为毒相关证据」
      │
      ▼
④ deepseek-v4-pro（当前激活模型）生成：
   - 改编后的完整剧本（全本）
   - 改编说明（记录改动点、逻辑调整、线索变更）
   - **截断检测：** 检查 API 返回的 `finish_reason`；若为 `"length"`（截断），前端醒目提示「改编内容可能被截断，建议分段改编或缩短改编范围」

**改编输出安全保障：**
- `max_tokens` 设置为 **80,000**（足够覆盖最大剧本全本输出）
- 检测 `finish_reason`：`"stop"` = 正常完成；`"length"` = 被截断，需提示
- 导出前自动校验：改编版幕数 = 原版幕数（除非改编指令明确要求增减），角色数量一致；若不一致，提示管理员核对
- 导出文件自动添加「改编版本 · 仅供内部使用」水印
      │
      ▼
⑤ 管理员预览对比（原版 vs 改编版，并排显示）
      │
      ▼
⑥ 确认导出（DOCX 或 PDF）
      │
      ▼
⑦ 操作日志写入（见下节）
```

#### 4.2.4 安全兜底机制

| 机制 | 实现方式 |
|------|---------|
| **仅管理员触发** | API 路由检查 `user.role === 'admin'`，否则 403 |
| **操作日志** | 每次改编记录：操作人、时间、剧本名、改编指令、改动摘要、导出文件路径 |
| **敏感本标记** | 剧本可标记 `is_sensitive`，敏感本改编需二次确认；日志永久保留 |
| **导出水印** | 导出文件自动添加「改编版本 · 仅供内部使用」水印 |

**操作日志表：**

```sql
adaptation_logs (
  id UUID PRIMARY KEY,
  operator_id UUID REFERENCES employees(id),
  script_id UUID REFERENCES scripts(id),
  adaptation_type VARCHAR(50),     -- element_replacement / plot_tweak / perspective_expand / manual_adapt
  instruction TEXT,                -- 管理员的改编指令原文
  changes_summary TEXT,            -- AI 生成的改动摘要
  output_file_path VARCHAR(500),   -- 导出文件路径
  created_at TIMESTAMP DEFAULT NOW()
)
```

**敏感本标记（初步判定框架）：**

以下类型剧本建议标记为敏感，管理员可根据本店情况调整：
- 涉及真实历史事件或政治人物
- 包含极端暴力、血腥细节的描述
- 涉及宗教信仰的争议性内容
- 改编后可能引发玩家不适的其他内容

标记由管理员手工设置，敏感本改编时强制显示风险提示，需二次确认。

**通用操作日志表：**

```sql
operation_logs (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES employees(id),
  action VARCHAR(50),              -- login / logout / query / permission_denied / model_switch / upload / delete
  detail TEXT,                      -- 操作详情（可包含脚本名、问题摘要等）
  ip_address VARCHAR(45),
  user_agent TEXT,
  created_at TIMESTAMP DEFAULT NOW()
)
```

```sql
scripts (
  id UUID PRIMARY KEY,
  name VARCHAR(255),
  version INTEGER DEFAULT 1,              -- 版本号，每次上传新文件自增
  -- 元数据字段（管理员录入，用于筛选和搜索）
  author VARCHAR(100),                    -- 作者/发行方
  genre VARCHAR(50),                      -- 类型：硬核/情感/恐怖/欢乐/阵营/其他
  player_count VARCHAR(20),              -- 人数：如 "6人"、"6-8人"
  act_count INTEGER,                     -- 幕数
  difficulty VARCHAR(20),               -- 难度：简单/中等/困难
  duration VARCHAR(20),                  -- 预计时长：如 "4-5小时"
  -- 安全相关
  is_sensitive BOOLEAN DEFAULT FALSE,
  sensitivity_note TEXT,                  -- 敏感原因
  -- 存储相关
  original_file_path VARCHAR(500),
  cached_text_path VARCHAR(500),          -- 解析后的缓存文本路径
  cache_checksum VARCHAR(64),             -- 缓存文件 MD5，用于检测损坏
  cached_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP
)
```

### 4.3 模块三：员工信息查询

**适用角色：** 员工 + 管理员

**权限设计（v1 两级）：**

```
员工（staff）
  → 只能查询自己的信息（姓名、职位、联系方式）
  → 只能查看自己的排班
  → 可以问：「我这周哪天上班？」

管理员（admin）
  → 可以查询所有员工信息
  → 可以管理排班（增删改）
  → 可以问：「全店本月排班统计」
  → 可以增删改员工档案

（组长 role → v2 规划）
```

**员工能问的典型问题：**
- "我这周哪天上班？"
- "本月我上了多少场？"

**管理员专属（员工无法问以下问题）：**
- "周六下午缺一个DM，谁能来？" → **隐私保护：员工不能看到其他人的排班**
- "小明的电话号码是多少？"
- "帮我统计本月每人上了多少场"
- 增删改员工档案、排班录入（通过管理界面，不走AI）

> **隐私保护说明：** 员工查询的范围严格限于自己。排班可见性是敏感数据，员工提问"谁能来"会被系统拒绝回答，提示"此操作需要管理员权限"。v2 如引入组长 role，可考虑给组长开放特定班次的排班可见性。

### 4.4 错误处理与用户提示设计

| 异常场景 | 前端提示 | 说明 |
|---------|---------|------|
| LLM 调用超时（>30s） | "AI 回答超时，请稍后重试" | 自动重试 1 次，仍失败则提示用户 |
| LLM 返回错误 | "AI 服务暂时不可用，管理员已收到通知" | 同时记录 `operation_logs` |
| 剧本解析失败 | "剧本格式不支持或已损坏，请联系管理员" | 前端显示失败原因（可选） |
| 权限不足（403） | "您没有权限执行此操作" | 不暴露具体原因 |
| 网络中断（前端检测） | "网络连接已断开，请检查网络后重试" | 前端轮询检测 |
| 磁盘空间不足 | 管理员后台醒目告警（不影响员工使用） | 80% 告警，95% 阻止新剧本上传 |

### 4.5 前端交互规范

> **背景：** 员工"大部分使用手机端"，而后端架构设计完善但前端交互几乎空白。本节补充手机端的核心交互规范，避免开发阶段返工。

#### 4.5.1 前端适配策略

**选型结论：响应式 Web + 基础 PWA 配置**

| 方案 | 结论 | 说明 |
|------|------|------|
| 响应式 Web（一套适配手机/PC） | ✅ 采用 | Next.js 默认支持，开发成本低，满足基本需求 |
| 独立移动端页面 | ❌ 不做 | v1 优先交付功能，不做独立 iOS/Android |
| PWA（添加到主屏幕） | ✅ 启用 | 类原生体验，Next.js 支持好，显著提升手机端留存 |

**PWA 基础配置（v1 实现）：**
- 添加到主屏幕快捷方式
- 启动画面（可复用登录页）
- 离线提示（"网络已断开，请检查网络后重试"）
- Service Worker 缓存基础资源

> **管理员端限定 PC 操作（v1）：** 剧本上传/改编工坊/员工管理/排班管理等复杂操作仅支持 PC 端。手机端仅提供基础查看能力（查看排班、查看剧本列表）。

| 问题 | 规范 |

| 规则 | 值 |

| 自动登出 | 30 分钟无操作后自动登出 |
| 登录页面 | 员工号 + 密码，内网 HTTPS 加密传输 |

#### 4.5.6 对话历史持久化

| 问题 | 规范 |
|------|------|
| 关闭浏览器再打开 | 对话历史保留（服务端 session） |

| 存储位置 | 服务端 session（而非 localStorage，确保多设备一致） |
| 历史上限 | 每个剧本保留最近 50 轮对话，超出后按压缩策略处理（见 3.3 节） |

#### 4.5.7 自签名证书手机端兼容说明

员工首次访问时需在浏览器中信任自签名证书：

| 设备/浏览器 | 信任方式 | 备注 |
|------------|----------|------|

| 企业微信/钉钉 | ⚠️ 可能有额外安全限制 | **需实际测试** |

> **建议：** 部署文档中补充手机端各浏览器的证书信任操作步骤截图，特别是 iPhone Safari（步骤最多）。上线前使用真机测试各浏览器兼容性。

## 5. 数据架构

### 5.1 数据存储总览

```
/data/
├── scripts/                           # 剧本存储
│   └── {script_id}/
│       ├── original.docx              # 原始文件（改编用）
│       ├── original.pdf               # 原始文件备份
│       └── parsed.txt                 # 解析后的纯文本缓存
│
└── exports/                           # 改编导出文件
    └── {script_id}/
        └── {timestamp}_adapted.docx

PostgreSQL: 员工数据（与剧本数据完全独立）
  ├── employees 表
  ├── schedules 表
  ├── adaptation_logs 表
  └── operation_logs 表              -- 通用操作日志（登录/登出/权限拒绝等）
```

### 5.2 剧本解析流程

```
原始文件（DOCX / PDF）
        │
        ▼
FastAPI 微服务（唯一 Python 依赖）
  ├── DOCX：python-docx 解析 XML，提取标题层级 + 正文
  └── PDF：PyMuPDF 提取文本 + 识别标题结构
        │
        ▼
标准化文本格式：
  【剧本名】
  【第一幕】
  （角色A剧本）
  ...
  （角色B剧本）
  【第二幕】
  ...
  【DM手册】
  ...
  【线索卡】
  ...
        │
        ▼
写入 /data/scripts/{id}/parsed.txt
```

**支持的文件格式与限制：**

| 格式 | 支持 | 限制说明 |
|------|------|----------|
| DOCX（Word） | ✅ | 标准 Office Open XML 格式；受密码保护的 DOCX 需先解除密码 |
| PDF（文本型） | ✅ | 通过 PyMuPDF 直接提取文本的 PDF |
| PDF（扫描图片型） | ❌ | 无 OCR 能力，图片型 PDF 需先通过外部工具（如 Tesseract）做 OCR 转换 |
| 加密/受密码保护文件 | ❌ | 上传前需解除密码保护 |
| TXT / HTML / 其他 | ❌ | v1 仅支持 DOCX 和 PDF；其他格式需转换为 PDF 或手工录入（v2 规划） |
| 特殊排版剧本 | ⚠️ | 大量嵌套表格、分栏排版、多列混排的剧本，解析质量可能下降，需人工核对 |

> **建议：** 入库前将所有剧本统一为 DOCX 格式（Word 可直接保存为 DOCX），是最稳定的解析来源。

**剧本入库端到端流程：**

```
① 管理员进入「剧本管理」→「添加剧本」
      │
      ▼
② 填写元数据表单
   【必填】剧本名称、原始文件（DOCX/PDF）
   【必填】人数、类型（硬核/情感/恐怖/欢乐等）
   【选填】作者、幕数、难度、预计时长
   【选填】敏感本标记 + 敏感原因
      │
      ▼
③ 上传文件 → 前端校验格式
   ├── 格式不支持 → 提示「请上传 DOCX 或 PDF 文件」
   ├── 文件过大（>50MB）→ 提示「文件超出大小限制」
   └── 格式通过 → 继续
      │
      ▼
④ FastAPI 解析（上传成功后自动触发）
   ├── 解析成功 → 写入 /data/scripts/{id}/parsed.txt
   │               记录 cache_checksum（MD5）
   ├── 解析失败 → 提示「剧本解析失败，请检查文件格式」，保留上传记录但不建立缓存
   └── 磁盘空间不足 → 阻止上传，提示管理员清理空间
      │
      ▼
⑤ 剧本上线
   → 员工可在剧本列表中搜索、筛选（按类型/人数/难度）
   → 首次被问时注入全本
```

**批量导入（v2 规划）：** 第一阶段暂不支持批量上传。若第一批 50-100 本剧本需入库，建议通过运维脚本将文件放入 `/data/scripts/` 目录后，在管理后台逐个"触发解析"。v2 补充批量上传入口。

  id UUID PRIMARY KEY,
  name VARCHAR(100) NOT NULL,

## 7. 部署架构

### 7.1 部署位置与配置

**店内现有服务器/电脑**（能跑 Docker 即可）

| 资源 | 最低配置 | 推荐配置 |
|------|---------|---------|
| CPU | 4核 | 8核+ |
| 内存 | 8GB | 16GB |
| 磁盘 | 256GB | 500GB+（剧本DOCX/PDF预计10-50GB） |
| 网络 | 千兆局域网 | 同上 |

### 7.2 访问方式

```
员工手机/电脑 ──►  https://192.168.x.x:3000  （内网 HTTPS）
                              │
                    Next.js Auth（内网账号登录，密码 HTTPS 加密传输）
                              │
                    无公网暴露 / 无需 VPN
```

> **HTTPS 要求：** 即使在内网，登录密码也不能明文 HTTP 传输。部署时使用自签名证书（mkcert 生成），浏览器首次访问时手动信任即可。无需购买商业证书。

**IP 地址管理：**

| 方案 | 配置方式 | 适用场景 |
|------|---------|---------|
| 固定 IP（推荐） | 在路由器/服务器设置静态 IP，如 `192.168.1.100` | 长期稳定使用 |
| DHCP + 路由器静态租约 | 在路由器 DHCP 设置中将该 MAC 地址分配固定 IP | 与固定 IP 效果相同 |
| mDNS / Bonjour（备选） | 服务器启动 `avahi-daemon`（Linux）或对应服务，访问 `https://dm-assistant.local:3000` | 手机/电脑需支持 mDNS；适合不想记 IP 的场景 |

> **IP 变更处理：** 若服务器 IP 变了，在路由器「静态 IP 分配」页面更新 DHCP 租约即可。员工端无需任何操作——建议在系统「帮助」页面注明当前访问地址，由管理员维护。

**员工首次访问自签名证书操作指引（部署文档附）：**
1. 浏览器访问 `https://192.168.x.x:3000`
2. 浏览器提示「连接不安全」→ 点击「高级」→「继续前往」
3. （Chrome）输入 `thisisunsafe`（页面任意位置，非地址栏）绕过警告
4. 登录后浏览器会记住证书，后续访问不再提示

### 7.3 数据备份策略

**备份范围：**

| 数据 | 位置 | 备份方式 |
|------|------|---------|
| PostgreSQL 数据库 | 容器内数据卷 | 每日凌晨 `pg_dump` 导出 SQL 文件，保留最近 30 天 |
| 剧本原始文件 | `/data/scripts/` | 随数据库备份一起打包，或 rsync 到第二块磁盘 |
| 剧本解析缓存 | `/data/scripts/*/parsed.txt` | **不备份**——可从原始文件重新解析生成 |
| 改编导出文件 | `/data/exports/` | 随数据库备份一起打包 |

**备份存储位置：**
- 最低要求：同一机器的第二块磁盘/NAS
- 推荐：每日备份推送至异地（NAS 或网盘）

**恢复测试：** 每季度执行一次备份恢复演练，验证备份文件完整性。

> v1 不实现自动备份脚本，由运维人员手动配置 crontab 或 NAS 同步任务。

**Token 消耗参考：**

> - 场景假设：剧本库规模100本，每本被问N次/天（首次问答约31K Token/次，后续问答约6K Token/次）
> - 表格中的 Token 数为**输入侧 Token**；输出 Token 约为输入的 5-10%（约3-10M/月），已计入成本估算

| 场景 | 日问答次数 | 月Token（输入） | 月成本（标准价） | 月成本（优惠价） | 说明 |
|------|-----------|----------------|---------------|---------------|------|

| **主力场景（500次/天）** | **500** | **~165M** | **¥120-140/月** | **¥30-40/月** | 每本约5次/天（与3.3节一致） |

> **剧本库扩展提示：** 剧本库越大，首次问答次数越多（每本首次被问时产生高Token消耗），月Token成本会随库容增长。缓存命中后，每次后续问答的增量成本极小。

| **合计** | **¥60-140/月** | 几乎可忽略 |

### v2 规划清单（归档待做项）

以下为 v1 完成后的优化方向，不影响 v1 交付：

| 建议项 | 优先级 | 说明 |
|--------|--------|------|
| 批量剧本导入 | **P1** | 批量上传、批量解析、批量入库；第一阶段痛点 |
| 改编撤销/回退 | **P1** | 基于 `version` 字段实现版本回退；防止改编失误 |

| 问答引用溯源 | **P2** | 回答时标注引用自哪一幕/哪个角色；提升答案可信度 |
| 剧本概览卡片 | **P2** | 快捷提问模板；降低员工使用门槛 |
| 流式输出断点续传 | **P3** | 网络中断后从断点恢复输出；体验优化 |
